# ESLint 9-to-10 upgrade with compat shims for legacy plugins

ESLint is upgraded from v9.39.5 to v10.7.0, with `@eslint/js` bumped to match. Three plugins that still call the removed `context.getFilename()` API â€” `eslint-plugin-react`, `eslint-plugin-jsx-a11y`, and `eslint-plugin-import` â€” are wrapped with `fixupPluginRules` from the new `@eslint/compat` package. `eslint-plugin-react-hooks` v7.1.1 already declares ESLint 10 peer support and is left unwrapped. The config passes cleanly (`npx eslint .` exits 0).

**Watch for:** Three plugins declare `peerDependencies` that cap at ESLint 9 (confirmed). `npm ls eslint` reports them as `invalid`. This works today because npm doesn't block installs on peer mismatches by default, and `fixupPluginRules` papers over the runtime API gap â€” but a future plugin release could introduce assumptions about ESLint 10 internals that the compat shim doesn't cover.

## High-level view

The migration strategy is minimal-surface: bump ESLint, add `@eslint/compat`, wrap the three offending plugins, ship. No rules were changed, no plugin versions were bumped, and no new lint capabilities were enabled. This keeps the blast radius small but leaves the project in a transitional state where three plugins are running against an ESLint version they don't officially support.

The peer-dependency violations are the primary risk. `eslint-plugin-import` (v2.32.0), `eslint-plugin-react` (v7.37.5), and `eslint-plugin-jsx-a11y` (v6.10.2) all declare `eslint: ^9` or lower as their peer. The compat shim handles the specific `context.getFilename()` â†’ `context.filename` rename that ESLint 10 enforced, but it doesn't guarantee full compatibility with other internal API changes across the major version boundary. CI will catch runtime failures, but silent behavioral drift (e.g., a rule silently skipping nodes due to an unshimmed API) would not surface.

## `fixupPluginRules` scope and the silent-failure mode

`fixupPluginRules` from `@eslint/compat` creates a proxy around the plugin object that intercepts rule `create()` calls and provides a shimmed context. The shim maps removed/renamed properties (`context.getFilename()` â†’ `context.filename`, `context.getSourceCode()` â†’ `context.sourceCode`, etc.) so legacy rules continue to function.

The failure mode (likely) is not a crash but a silent skip: if a future ESLint 10.x release changes how rule visitors are registered or how the AST is provided, a shimmed plugin rule could silently stop matching nodes. The linter would still exit 0, but rules would no longer fire. This is particularly relevant for `eslint-plugin-import`, whose `import/order` rule inspects module graph information that comes from ESLint's module resolver infrastructure â€” an area that's been actively refactored across major versions.

A lightweight guard: run `eslint --print-config src/someFile.tsx` in CI and assert that the expected rules are present and set to their expected severity. This catches the scenario where a plugin silently drops rules after an update.
